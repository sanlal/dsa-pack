Object-Oriented Programming

What is OOP?

Object-Oriented Programming is a way to write code. It puts the data and the functions that change that data together in one place, called an object. The blueprint for the object is called a class.

In one line.  OOP organises code around objects — small, self-contained units that hold their own data and enforce their own rules. The point is to keep change small: when something changes, only one class should need to change.

The four pillars of OOP

🔒

Encapsulation

Keep data private. Expose only methods that validate and change it.

🏠

Inheritance

A new class reuses everything from an existing one. Models “is-a”.

🎬

Polymorphism

One method name does the right thing for each object type at runtime.

✨

Abstraction

Show what something does. Hide the messy details of how it does it.

Why OOP exists

The problem with procedural code, and the day a manager broke 30 of your functions.

OOP (Object-Oriented Programming) is a way to write code. It puts data and the functions that change that data together in one box. The box is called an object. The blueprint for the box is called a class.

The world before OOP

Programmers used to write code in a procedural style. Data sat in one place. Functions sat in another. Every function had to take the data as parameters, do its work, and return a result.

This was fine for a 100-line script. It became painful for a 100,000-line app. Three reasons:

  • Data and behaviour drift apart. A change in the data shape forces edits in dozens of unrelated functions.
  • No rules are enforced. Any code can write any value anywhere. There is no central place that says “a price can never be negative”.
  • The mental model breaks down. You have to keep the whole program in your head to be sure you didn't miss a function.

The Swiggy story

Imagine you are building Swiggy without OOP. You need to track 1,000 restaurants. Each has a name, address, rating, phone, opening hours, menu, and an open/closed flag.

You start with loose variables:

name_1   = "Pizza Hut"
addr_1   = "Koramangala"
rating_1 = 4.2
is_open_1 = True
# ... and so on for 1,000 restaurants

Then your place_order function takes a long list of parameters:

def place_order(name, address, phone, is_open, menu, items, total):
    if not is_open:           return "closed"
    if total < 100:           return "min order is 100"
    return "order placed"

Now your product manager says: “Please add a WhatsApp number to every restaurant.” You add whatsapp_1 through whatsapp_1000. You update place_order to accept and use it. Then show_menu. Then send_receipt. Then notify_owner. You update 30 functions. You miss one. The app crashes in production.

The OOP fix

OOP says: write one class called Restaurant that lists everything a restaurant has and everything it can do. Make 1,000 objects from that blueprint. Each carries its own copy of the data.

class Restaurant:
    def __init__(self, name, address, phone):
        self.name      = name
        self.address   = address
        self.phone     = phone
        self.is_open   = True
        self.menu      = []

    def place_order(self, items):
        if not self.is_open: return "closed"
        total = sum(i.price for i in items)
        if total < 100:      return "min order is 100"
        return "order placed"

When the manager says “add WhatsApp”, you edit the class once. All 1,000 restaurants get the field for free. The blast radius of the change is one file.

What OOP gives you

  1. Smaller changes. One change stays inside one class.
  2. Real rules. A class can refuse to enter a bad state. Negative balances become impossible.
  3. Code that groups by “thing”. Everything about restaurants lives in Restaurant.py.
  4. Parallel teamwork. Three engineers can work on Order, Payment, and Notification without colliding.
  5. Vocabulary that scales. “The order needs a payment” is easier to discuss than passing amounts around.
The big idea

OOP is not really about classes, pillars, or fancy words. It is about keeping change small. Every other concept — encapsulation, inheritance, polymorphism, composition — is a tool for that goal.

Class vs Object

A blueprint on paper vs the actual building made from it.

Class = a blueprint. It lists what a thing has and what it can do. Object = one real thing made from that blueprint, with its own values.

The cookie-cutter analogy

Picture a metal cookie cutter shaped like a star. By itself the cutter is not a cookie. You can't eat it. But you can press it into dough a hundred times and get a hundred star-shaped cookies. Each cookie is real. Each one can have its own colour of icing.

The class is the cutter. The objects are the cookies.

Two objects, same class, different state

Each object has its own copy of the fields. Changing one does not touch the others.

class Restaurant:
    def __init__(self, name, rating):
        self.name   = name
        self.rating = rating

pizza_hut = Restaurant("Pizza Hut", 4.2)
dominos   = Restaurant("Dominos",   4.0)

pizza_hut.rating = 4.5             # only changes pizza_hut
print(dominos.rating)              # still 4.0

Object vs instance — same thing

Engineers say object in casual talk and instance in formal talk. They mean exactly the same. “Pizza Hut is an instance of the Restaurant class” is identical to “Pizza Hut is a Restaurant object”.

Quick mental check

If a class definition prints something or runs logic by itself, that's a bug. A class should declare structure only. Logic runs when you call a method on an object.

What makes a method different from a function?

Notice add() changes items and total without you passing them in. They are already inside the object. Method access to its own fields is the key thing separating OOP from procedural code.

  • A function stands alone and works only on its parameters.

Constructor — the birth of an object

The special method that runs once, when the object is created.

Constructor = a special method with one job: to set up the new object so it starts life in a valid state.

The sign-up form analogy

When a new restaurant joins Swiggy, the owner fills a form: legal name, address, FSSAI number, opening hours, bank account. Until the form is complete, the restaurant does not exist in Swiggy's system. The form is the constructor.

The “no half-baked object” rule

Without a constructor, someone could create a Restaurant with no name and no address. Then they call pizza_hut.place_order(...) and the app crashes. The constructor exists to make sure that by the time the object exists, it is valid.

Validate inside the constructor

class Restaurant:
    def __init__(self, name, address, fssai):
        if not name:               raise ValueError("name required")
        if not address:            raise ValueError("address required")
        if len(fssai) != 14:       raise ValueError("invalid FSSAI")
        self.name    = name
        self.address = address
        self.fssai   = fssai
Common beginner trap

You write Restaurant(String name) and now new Restaurant() fails to compile. The free no-argument constructor is taken away the moment you declare any constructor. If you want both, declare both.

Senior habit

Treat the constructor like the bouncer at a club. If the inputs are bad, throw them out at the door. Every line of code after the constructor should be able to trust that the object is in a valid state.

this and self — the word “I”

How a method refers to its own object.

this (C++/Java) and self (Python) both mean “the current object — the one I am running inside right now”. It is how a method talks about itself.

The “I, Pizza Hut” analogy

When Swiggy's code calls pizza_hut.place_order(items), the method runs inside Pizza Hut. While running, the method needs a way to say “I, Pizza Hut, am the one taking this order”. That word “I” is self. A second later when the same method runs on Dominos, self means Dominos.

The most common use — disambiguating names

class Driver:
    def __init__(self, name, rating):
        self.name   = name     # "my name (field) = the parameter name"
        self.rating = rating
class Driver {
    String name;
    double rating;
    Driver(String name, double rating) {
        this.name   = name;
        this.rating = rating;
    }
}
class Driver {
public:
    string name;
    double rating;
    Driver(string name, double rating) {
        this->name   = name;
        this->rating = rating;
    }
};

Other places this shows up

  • Method chaining. Return this from a setter to chain: cart.add(x).add(y).checkout().
  • Passing self to another object. orderQueue.submit(this) hands the current object to a helper.
  • Identity comparison. if (other == this) return true; checks “is this the same exact object?”

Python is different

In Python you always write self as the first parameter of every instance method. Java and C++ hide it — the language passes this behind the scenes so the body never declares it.

Convention

The name self in Python is not reserved — you could call it this. Don't. Every Python codebase uses self; deviating confuses your teammates and your tools.

Static vs Instance

One for all, or one per object. A surge multiplier vs the fare of one trip.

An instance member belongs to each object. A static member belongs to the class itself — it is shared by every object.

The Uber surge analogy

Every trip has its own fare — that fare is an instance field. But the current surge multiplier (1.0x, 1.5x, 2.0x) is the same for every rider in the area at this moment — that's a static field.

More real-life pairs

How to declare static members

When to use static

  • Constants. MAX_GROUP_SIZE = 1024, GST_PERCENT = 18.
  • Counters. Total users created, total API calls so far.
  • Utility methods. Math.sqrt(x), Integer.parseInt(s).
  • Factory methods. User.fromJson(data).
Beware of static overuse

Static is global state in disguise. Tests interfere with each other. Multi-threading needs locks. Hidden coupling makes refactoring scary. Use it for constants and counters, not for business logic.

Object lifecycle — from birth to death

Allocation, construction, use, destruction.

Every object goes through four phases: allocate memory, construct, use, and destroy. Languages handle the last step very differently.

The four phases

  1. Allocate. The system reserves memory for the new object's fields.
  2. Construct. Your constructor runs. Parent class first (if any), then child. Fields get their initial values.
  3. Use. Callers call methods. The object reads and updates its own fields. Most of the object's life.
  4. Destroy. Memory is reclaimed. C++ does this exactly when the object goes out of scope. Java and Python use a background garbage collector that runs at unpredictable times.

An Uber trip's lifecycle

You open the Uber app and book a ride. A new Trip object is born (steps 1 and 2). For the next 25 minutes the trip lives — driver updates location, fare ticks up, you chat (step 3). You reach home, the trip ends, the object's job is done; memory is freed (step 4).

Destructor (C++) vs Garbage Collection (Java/Python)

C++ — you control the moment (RAII)

class FileLogger {
    ofstream file;
public:
    FileLogger(string path) { file.open(path); }
    ~FileLogger() { file.close(); }   // runs when object dies
};

{
    FileLogger log("x.log");
    log.write("hi");
}   // log dies HERE — file closed for sure

Java / Python — the GC controls it

# Python
with open("x.log") as f:
    f.write("hi")             # closed when block ends

// Java
try (FileLogger log = new FileLogger("x.log")) {
    log.write("hi");
}                             // close() called here
Common bug: don't call overridable methods from a constructor

If a parent constructor calls self.process() and a child overrides process, the child's version runs — before the child's own fields are initialised. You will read None or undefined values. Classic, painful bug.

The one rule that matters most

If your object opens something — a file, socket, DB connection, lock — make sure there is a clear, named way to close it. Garbage collection frees memory, not resources.

Pillar 1 — Encapsulation

Hide the wires. Show only the buttons.

Encapsulation means: keep the fields private, expose methods that change them, and check the inputs inside those methods. The outside world cannot put your object into a broken state.

The ATM analogy

An ATM has cash inside. You cannot open the back and grab notes. You press buttons (methods): insert card, enter PIN, choose amount. The machine checks the rules — right PIN? enough balance? under daily limit? — and only then dispenses cash.

Apps that are heavily encapsulated

  • Uber. You cannot set the driver's location to your house. You call requestRide(); Uber checks rules and assigns the nearest driver.
  • Paytm. You cannot directly add ₹1,000 to your wallet. You call addMoney(1000, source); Paytm verifies the source, deducts from your bank, then credits the wallet.
  • Instagram. You cannot set a post's like-count to 1 million. You call like() once per user. The count goes up by exactly 1.

No encapsulation vs encapsulated

No encapsulation

class BankAccount: pass

me = BankAccount()
me.balance = -50000   # theft, undetected

No rules, no safety, no audit trail.

Encapsulated

class BankAccount:
    def __init__(self, opening):
        self.__balance = opening
    def deposit(self, amt):
        if amt <= 0:
            raise ValueError("amt <= 0")
        self.__balance += amt

Field hidden. Method validates. Negative deposits rejected at the door.

A real-looking BankAccount in all three languages

class BankAccount:
    def __init__(self, opening):
        if opening < 0: raise ValueError("opening < 0")
        self.__balance = opening

    def deposit(self, amount):
        if amount <= 0: raise ValueError("amount <= 0")
        self.__balance += amount

    def withdraw(self, amount):
        if amount <= 0:               raise ValueError("amount <= 0")
        if amount > self.__balance:   raise ValueError("insufficient funds")
        self.__balance -= amount

    @property
    def balance(self): return self.__balance
public class BankAccount {
    private double balance;

    public BankAccount(double opening) {
        if (opening < 0)
            throw new IllegalArgumentException("opening < 0");
        this.balance = opening;
    }
    public void deposit(double amt) {
        if (amt <= 0) throw new IllegalArgumentException("amt <= 0");
        balance += amt;
    }
    public void withdraw(double amt) {
        if (amt <= 0)      throw new IllegalArgumentException("amt <= 0");
        if (amt > balance) throw new IllegalStateException("insufficient funds");
        balance -= amt;
    }
    public double getBalance() { return balance; }
}
class BankAccount {
    double balance_;
public:
    explicit BankAccount(double opening) {
        if (opening < 0) throw runtime_error("opening < 0");
        balance_ = opening;
    }
    void deposit(double amt) {
        if (amt <= 0) throw runtime_error("amt <= 0");
        balance_ += amt;
    }
    void withdraw(double amt) {
        if (amt <= 0)        throw runtime_error("amt <= 0");
        if (amt > balance_)  throw runtime_error("insufficient funds");
        balance_ -= amt;
    }
    double balance() const { return balance_; }
};

Getter / setter discipline

A class that's all getters and setters has no encapsulation — it's a public struct in disguise. Prefer behaviour methods over raw setters.

Bad styleGood style
acc.setBalance(acc.getBalance() + 100)acc.deposit(100)
order.setStatus("CANCELLED")order.cancel(reason)
user.setLoggedIn(true)user.login(password)
The senior heuristic

If you can think of a way someone could mis-use your class, write a method that makes that mis-use impossible, then make the field private.

Pillar 2 — Inheritance

A child class reuses everything from the parent and adds its own bits.

Inheritance means a new class (child) is built on top of an existing class (parent). The child gets every public/protected field and method of the parent for free, and can add or override its own.

The YouTube account analogy

On YouTube, everyone starts as a User — you have an email, watch history, like list. The moment you click “Create a channel”, you become a Channel. A channel is still a user (you keep your email and history), but now you also have extras: a channel name, a list of uploads, a subscriber count.

We say Channel is a User. The “is a” test is how you decide whether inheritance is the right tool.

The diamond problem

Two grandparents become one. Which copy wins?

When a class inherits from two parents who both inherit from the same grandparent, the grandparent's fields appear twice. Each language solves this differently. This is the classic OOP interview question.

The shape

          Device
         /      \
     Printer   Scanner
         \      /
        PhotoCopier   «-- inherits from both

Now PhotoCopier seems to have two copies of every Device field. Which power button do you press? That's the diamond problem.

C++Mark grandparent as virtual: class Printer : virtual public Device. Without it, you really do get two copies.

Pillar 3 — Polymorphism

One word, many forms. The same call does the right thing for each object.

Polymorphism means “one name, many forms”. You call player.play(); if the object is a Movie it streams, a Song it plays audio, a Podcast it resumes. Same call, different behaviour, and the caller doesn't have to know which type it is.

The Netflix play button

The big red Play button runs play(). But Netflix has Movie, Series, Documentary, StandUp, Trailer. Each one needs to start differently — a movie resumes from where you left off, a series resumes the last episode, a trailer starts at 0. The app calls currentItem.play(); the correct version runs based on the real type.

Why this is powerful

Tomorrow Netflix adds InteractiveShow. They write one new class with its own play(). The Play button code does not change. Not a single line.

Two flavours

Compile-time (static)

Compiler decides which method to call by looking at the arguments. Fixed before runtime.

Mechanism: method overloading (topic 15).

Runtime (dynamic)

Decided while the program runs, based on the actual object type.

Mechanism: method overriding (topic 16). Powers the Netflix play button.

How runtime dispatch works (vtable)

Every class with virtual methods carries a hidden table of function pointers. Each object stores a pointer to its class's vtable. When you call shape->area(), the compiler emits “look up area in the vtable, then call it”. One indirection — cheap, but why C++ asks you to opt in via virtual.

Pillar 4 — Abstraction

Show what it does. Hide how it does it.

Abstraction means: expose a simple contract; hide the messy details. The caller depends on the contract. Tomorrow the implementation can change completely — the caller doesn't notice.

The car steering wheel analogy

Turn the wheel left, the car goes left. You don't know — or care — whether the steering is hydraulic, electric, or steer-by-wire. The wheel is the abstraction; the rest is hidden.

Apps layered with abstraction

  • Swiggy “Place Order” button. Hidden: the database write, SMS to the restaurant, delivery partner API, payment gateway dance, fraud check.
  • Paytm UPI. Hidden: NPCI routing, ICICI debits, HDFC credits, settlement batches.
  • YouTube upload. Hidden: transcoding to 480p/720p/1080p/4K, thumbnails, copyright fingerprinting, CDN propagation.
  • Google search. Hidden: crawl of trillions of pages, inverted index, ranking with hundreds of signals.

Encapsulation vs Abstraction

Encapsulation

Implementation angle. “I'll keep these fields private and validate every change.”

Abstraction

Design angle. “I'll expose only what callers need. The rest is my problem.”

Abstract class

A class that's half built — some methods done, some left for children.

An abstract class is a class you cannot create objects of directly. It declares some methods without implementation (children must fill them) and may have some methods with implementation (children inherit them as-is).

The Ola Vehicle analogy

Ola has Bike, Auto, Car. Each is a Vehicle. But “Vehicle” by itself is not a thing you can ride. You always say new Bike() or new Car(). Vehicle can hold shared code: start_trip() and end_trip() work the same for all. But calculate_fare() differs — bikes cost less than cars — so Vehicle declares it abstract.

Interface

A pure contract — “if you claim to do X, here are the methods you must have”.

An interface is a list of methods that a class promises to implement. It says “can-do”, not “is-a”. A class can implement many interfaces.

The “Payable” analogy

Swiggy, Zomato, Amazon, BookMyShow, IRCTC. Different apps, different products. But every one accepts a payment. Define an interface Payable:

Now Swiggy's Order, Amazon's Order, BookMyShow's Booking, IRCTC's Ticket all implement Payable. Different classes, same can-do behaviour.

Composition — strong has-a

The whole owns the part. Destroy the whole, the part dies too.

Composition means: one object holds another and controls its lifetime. Build a Car, it builds an Engine inside. Crash the car, the engine is gone too.

The Zomato Order analogy

A Zomato Order object owns:

  • An OrderId
  • A list of OrderItems
  • A Bill with breakdown (subtotal, tax, packaging, delivery)
  • A DeliveryAssignment

If you cancel the order and Zomato deletes the Order, all of those die with it. They existed only because that order existed.

Association & Dependency

“Uses-a” and “knows-of” — the loosest links.

Association = two independent classes know each other (peer link). Dependency = one class briefly uses another, often as a method parameter.

Association — Uber's Driver and Rider

Driver and Rider are completely separate classes. They have their own data and lifecycles. During a trip they are associated — the Trip object holds references to both. After the trip, the association ends.

        self.driver = driver    # association
        self.fare   = 0

    def end(self, fare):
        self.fare = fare

“Favour composition over inheritance.” Said by every senior, ever.

Equality & hashing

When are two objects “the same”?

Out of the box, a == b on objects compares memory addresses, not contents. To compare contents, override equals (Java), operator== (C++), or __eq__ (Python). And every time you override equality, you must also override hash. Otherwise hash maps break silently.

The Swiggy Order analogy

Two Order objects in memory, both with id SW-12345. Are they equal?

  • By default: no, they live at different addresses.
  • What you want: yes, because the order ID is the same.

The equals + hashCode contract

If two objects are equal, their hash codes must be equal. Otherwise, HashMap looks in the wrong bucket and “loses” them. The bug is silent.

class Order:
    def __init__(self, id, total):
        self.id, self.total = id, total

    def __eq__(self, other):
        return isinstance(other, Order) and self.id == other.id

    def __hash__(self):
        return hash(self.id)        # MUST be consistent with __eq__

a = Order("SW-1", 500)
b = Order("SW-1", 500)
print(a == b)                       # True
s = {a}
print(b in s)                       # True — thanks to consistent hash

    template<> struct hash<Order> {
        size_t operator()(const Order& o) const noexcept {
            return hash<string>()(o.id);
        }
    };
}

The five rules equals must satisfy

  1. Reflexive. x.equals(x) is true.
  2. Symmetric. If x.equals(y), then y.equals(x).
  3. Transitive. If x.equals(y) and y.equals(z), then x.equals(z).
  4. Consistent. Repeated calls return the same result.
  5. Null-safe. x.equals(null) is false, never throws.
The classic silent bug

Override equals but forget hashCode. Put 100 orders into a HashSet. Call set.contains(orderCopy) — returns false even though an equal order is in there. Always override them together.

Shallow vs Deep copy

Are you copying the box, or also the things inside the box?

A shallow copy duplicates the object's top-level fields. References inside still point to the same shared things. A deep copy recursively duplicates everything.

The WhatsApp chat backup analogy

You have a chat that contains messages and photos.

  • Shallow copy — copy the chat folder, but the photos inside are shortcuts to the originals. Delete an original photo, the copy's shortcut breaks.
  • Deep copy — copy the chat folder and all the actual photos. Two independent copies.

The bug it causes

import copy

original = {"items": ["apple", "banana"]}
shallow  = copy.copy(original)
deep     = copy.deepcopy(original)

original["items"].append("mango")

print(shallow["items"])     # ['apple', 'banana', 'mango']  « shared!
print(deep["items"])        # ['apple', 'banana']            « safe

Code in each language

import copy

chat_shallow = copy.copy(chat)         # top-level only
chat_deep    = copy.deepcopy(chat)     # full recursive clone
class Chat {
    List<Message> messages;

    Chat(Chat other) {                          // shallow
        this.messages = other.messages;
    }

    static Chat deepCopy(Chat other) {          // deep
        Chat c = new Chat();
        c.messages = new ArrayList<>();
        for (Message m : other.messages)
            c.messages.add(new Message(m));
        return c;
    }
}

    vector<Message*> messages;

    Chat(const Chat& other) : messages(other.messages) {}   // shallow

Shortcut

If your object holds only immutable things (strings, numbers, tuples, immutable value objects), shallow copy is safe and fast — nothing can be mutated through a shared reference.

Immutability — objects that never change

Thread-safe for free. Impossible to corrupt.

An immutable object's fields cannot be changed after construction. Need a “modified” version? Create a new object. Java String, Python str and tuple, and most value object types work this way.

The currency note analogy

A ₹500 note is immutable. You cannot turn it into ₹100 by scribbling on it — you'd just ruin it. To pay ₹100, you exchange the ₹500 for new notes.

How to make a class immutable

  1. Make all fields private final (Java) / const (C++) / set once in __init__.
  2. No setters. Only getters.
  3. If a field is itself mutable (like a List), return a copy from the getter.
  4. For a “modified” version, return a new instance.

Money class — the classic example

from dataclasses import dataclass

@dataclass(frozen=True)         # frozen = immutable
class Money:
    paise: int                  # INR as paise (avoid float bugs)

    def plus(self, other):  return Money(self.paise + other.paise)
    def minus(self, other): return Money(self.paise - other.paise)
    def times(self, n):     return Money(self.paise * n)

m1 = Money(50000)               # ₹500
m2 = m1.plus(Money(25000))      # ₹750 — new object
public final class Money {
    private final long paise;
    public Money(long p) { this.paise = p; }
    public long value() { return paise; }
    public Money plus(Money o)  { return new Money(paise + o.paise); }
    public Money minus(Money o) { return new Money(paise - o.paise); }
}
class Money {
    const long paise;
public:
    explicit Money(long p) : paise(p) {}
    long value() const           { return paise; }
    Money plus(const Money& o)  const { return Money(paise + o.paise); }
    Money minus(const Money& o) const { return Money(paise - o.paise); }
};

When to use

  • Value objects. Money, dates, coordinates, IDs, configs.
  • Domain events. “OrderPlaced”, “PaymentReceived” — facts don't change.
  • Map keys. Mutation breaks lookup.
  • Multi-threaded code. Safe to share with zero locking.
Senior preference

Make things immutable by default. Add mutability only where you must. Most production bugs come from unexpected mutation; immutability eliminates a whole category at the design level.

Coupling & cohesion

“Low coupling, high cohesion” — the goal of every class.

Coupling = how much one class depends on another. Cohesion = how related the things inside one class are. Good design = low coupling, high cohesion.

What they mean

Coupling (between classes)

Low = classes barely know each other.

High = classes are tangled. Change one, three break.

Aim: low.

Cohesion (inside one class)

High = everything in the class is about the same thing.

Low = the class does five unrelated jobs (the “God class”).

Aim: high.

The Swiggy example

  • High cohesion (good): RestaurantRepository handles only find/save/delete restaurants.
  • Low cohesion (bad): SwiggyUtils doing logging, distance, currency formatting, SMS. Five jobs, no theme.
  • Low coupling (good): checkout depends on PaymentGateway interface, not RazorpayClient.
  • High coupling (bad): checkout directly creates a RazorpayClient, reads its private fields.

The cure

# Before — tightly coupled to a specific SDK
class Checkout:
    def __init__(self):
        self.gw = RazorpayClient(api_key="...")

# After — depends on an abstraction
class Checkout:
    def __init__(self, gateway: PaymentGateway):
        self.gw = gateway
Shortcut

If a class does one thing well and depends only on small contracts of other classes, you've nailed both. Most of SOLID is just “how to lower coupling and raise cohesion”.

Common pitfalls

The traps every junior OOP coder falls into. Avoid all of them.

A shortlist of OOP traps. Read once. Look again before every code review.

Avoid

  • God class. One class with 40 methods doing five unrelated jobs.
  • Public fields. No encapsulation, no validation.
  • Inheritance for code reuse. If not “is-a”, compose.
  • Deep inheritance trees. Five levels usually means missed composition.
  • Override that removes behaviour. Classic LSP violation.
  • Calling overridable methods from a constructor.
  • Override equals, forget hashCode. Objects get lost in HashMap.
  • Forgetting virtual ~Base() in C++. Memory leak.
  • Returning the internal list from a getter. Anyone can mutate.
  • Static everywhere. Hurts testing and threading.
  • Big constructors that “do work”. Network, DB calls. Constructor should set up state.
  • Catching generic Exception. You hide bugs.

Do

  • Small classes. One reason to exist.
  • Private fields, behaviour methods. deposit(), not setBalance().
  • Program to interfaces. Depend on List, not ArrayList.
  • Compose first. Inherit only on clean “is-a”.
  • Immutable value objects. Money, Date, Point.
  • Honour equals/hashCode contract.
  • Use @Override / override.
  • Inject dependencies via constructor.
  • Validate in the constructor.
  • Write toString / __str__.
  • Throw specific exceptions. InsufficientFunds, not RuntimeException.
  • Use dataclass-like helpers. Python @dataclass, Java record.

Quick-fire interview Q&A

20 short questions, one-line answers, plus the killer 30-second pitch.

Read once. Read again on the morning of the interview.
  1. What is a class? A blueprint. Defines fields and methods. By itself does nothing.
  2. What is an object? A real instance created from a class. Each has its own state.
  3. What are the four pillars of OOP? Encapsulation, Inheritance, Polymorphism, Abstraction.
  4. What is encapsulation? Keep state private, expose validated behaviour methods.
  5. Abstraction vs encapsulation? Abstraction = expose only what's needed (design). Encapsulation = hide implementation behind methods (code).
  6. What is inheritance? Child reuses fields and methods of parent. Models “is-a”.
  7. What is polymorphism? One name, many forms. Same call resolves differently based on actual type.
  8. Compile-time vs runtime polymorphism? Overloading (compiler) vs overriding (runtime).
  9. Abstract class vs interface? Abstract class: state + partial behaviour, single inheritance, “is-a”. Interface: pure contract, multiple, “can-do”.
  10. Can we instantiate an abstract class? No.
  11. Java multiple inheritance? Not for classes. Yes for interfaces.
  12. Diamond problem? Two parents share grandparent; child gets two copies. C++ uses virtual inheritance; Java bans for classes; Python uses MRO.
  13. Method overloading? Same name, different parameters, compile time.
  14. Method overriding? Child redefines parent's method with same signature, runtime.
  15. Composition? Strong has-a. The whole owns the part.
  16. Aggregation? Weak has-a. The part survives the whole.
  17. Why composition over inheritance? Loose coupling; easier testing; mix and swap; runtime flexibility.
  18. equals/hashCode contract? If equal, hash codes must be equal. Otherwise hash collections break.
  19. Immutability? Object never changes. Thread-safe, impossible to corrupt, safe as map keys.
  20. Coupling and cohesion? Coupling = how much classes depend. Cohesion = how related things inside a class are. Aim: low coupling, high cohesion.
The killer 30-second pitch for “what is OOP?”

“OOP organises code around objects — bundles of data and the methods that act on them. The four pillars work together: encapsulation hides state, inheritance reuses for clean is-a, polymorphism swaps behaviour at runtime, abstraction stabilises the contract. The point is to keep change small: when a requirement shifts, only one class should need to change. In practice I lean heavily on composition for everything that isn't a clean is-a, and I prefer immutable value objects wherever possible.” Then pivot to a concrete example from a system you've built.

SOLID — the five principles every senior dev lives by

Five rules that turn good OOP into great OOP. Coined by Uncle Bob. Asked in 80% of senior interviews.

SOLID is an acronym for five design principles that make OOP code easier to change, easier to test, and harder to break. They are not laws. They are habits.

The five letters at a glance

LetterNameOne-line rule
SSingle ResponsibilityA class should have one reason to change.
OOpen / ClosedOpen for extension, closed for modification.
LLiskov SubstitutionA child should be usable wherever its parent is, without surprises.
IInterface SegregationMany small interfaces beat one fat interface.
DDependency InversionDepend on contracts, not on concrete classes.

S — Single Responsibility Principle (SRP)

A class should do one thing, and have one reason to be edited. If you find yourself saying “this class does X and Y”, that is two classes pretending to be one.

Swiggy example. An Order class that also sends emails, also writes to the database, and also calculates tax violates SRP three times. Split it into Order (data + business rules), OrderRepository (database), OrderEmailer (notifications), TaxCalculator (tax).

# BAD — one class, four reasons to change
class Order:
    def total(self): ...           # business rule
    def save_to_db(self): ...      # storage concern
    def send_email(self): ...      # notification concern
    def calculate_gst(self): ...   # tax rule (changes when govt rules change)

# GOOD — one class, one job
class Order:               def total(self): ...
class OrderRepository:     def save(self, order): ...
class OrderEmailer:        def confirm(self, order): ...
class TaxCalculator:       def gst(self, order): ...

O — Open / Closed Principle (OCP)

You should be able to add new behaviour without editing existing code. The cure is polymorphism: define a small contract and add new classes that implement it.

Uber example. When a city launches, you must add a new pricing rule. Without OCP, you edit a huge if/else inside calculatePrice and risk breaking the seven cities that already work.

# BAD — have to edit this function every time a city launches
def calculate_price(city, km, minutes):
    if city == "bangalore":  return 50 + 12*km
    elif city == "delhi":    return 60 + 14*km
    elif city == "mumbai":   return 70 + 15*km
    elif city == "chennai":  return 55 + 13*km   # add — risky edit

# GOOD — add a class. Don't touch existing ones.
class PricingStrategy:                         # the closed contract
    def price(self, km, minutes): ...
class BangalorePricing(PricingStrategy): ...
class DelhiPricing(PricingStrategy):     ...
class ChennaiPricing(PricingStrategy):   ...   # new file, zero edits

L — Liskov Substitution Principle (LSP)

If Bird is the parent of Sparrow, then anywhere that uses a Bird should be happy if you hand it a Sparrow instead. No surprises. No new exceptions. No weaker behaviour.

The classic violation. A Square that extends Rectangle seems natural — a square is a rectangle, right? But if your Rectangle.setWidth changes width independently of height, your Square.setWidth has to also change the height to keep it square. Anyone using a Rectangle and calling setWidth(5) expects only width to change. The Square breaks that expectation.

# BAD — child breaks parent's contract
class Rectangle:
    def __init__(self, w, h): self.w, self.h = w, h
    def set_width(self, w):   self.w = w        # only width changes

class Square(Rectangle):
    def set_width(self, w):                     # surprise! also changes height
        self.w = self.h = w

def stretch(rect):
    rect.set_width(10)
    assert rect.h == 5                          # passes for Rectangle, FAILS for Square

The fix is composition or a separate Square class with its own contract.

I — Interface Segregation Principle (ISP)

Don't force a class to implement methods it doesn't need. Split fat interfaces into focused ones.

WhatsApp example. A fat Message interface with send(), edit(), delete(), forward(), react(), schedule(), pin() forces every message class to implement all seven, even a simple system message that can only be send()'d.

# BAD — one fat interface
interface Message {
    send(); edit(); delete(); forward(); react(); schedule(); pin();
}
class SystemMessage implements Message {
    edit()    { throw new UnsupportedOperationException(); }   // smell
    delete()  { throw new UnsupportedOperationException(); }
    // ...etc
}

# GOOD — small interfaces, mix as needed
interface Sendable    { send(); }
interface Editable    { edit(); }
interface Reactable   { react(); }
class TextMessage    implements Sendable, Editable, Reactable { ... }
class SystemMessage  implements Sendable { ... }                  // clean

D — Dependency Inversion Principle (DIP)

High-level code (your business rules) should not depend on low-level code (database, payment gateway). Both should depend on an abstraction in between.

Why this matters. If OrderService directly creates new RazorpayPayment(), then switching to Stripe means editing OrderService. Worse, in a unit test you can't replace Razorpay with a fake. With DIP, OrderService depends on PaymentGateway (an interface). You pass in whichever implementation you want.

# BAD — high-level depends on low-level
class OrderService:
    def __init__(self):
        self.payment = RazorpayPayment()       # locked-in
    def checkout(self, order):
        return self.payment.charge(order.total)

# GOOD — both depend on PaymentGateway
class PaymentGateway:                          # the contract
    def charge(self, amount): ...

class RazorpayPayment(PaymentGateway): ...
class StripePayment(PaymentGateway):   ...
class FakePayment(PaymentGateway):     ...     # for tests

class OrderService:
    def __init__(self, gateway: PaymentGateway):
        self.payment = gateway                 # injected
    def checkout(self, order):
        return self.payment.charge(order.total)

How to remember SOLID in one breath

  1. Single — one class, one job.
  2. Open — add new code, don't edit old.
  3. Liskov — children must keep their parent's promises.
  4. Interface — small contracts, not god-interfaces.
  5. Dependency — depend on shapes, not on specific brands.
How interviewers ask this

“Tell me about SOLID.” or “Pick one principle and give a real example where you applied it.” Always have one short story per letter. SRP + DIP are asked the most.

Common mistake

People treat SOLID as five separate gods. They are one idea seen from five angles: keep change small and isolated. SRP is about who changes a class. OCP is about what edits are needed. LSP, ISP, DIP are all about how loose your contracts are.

Design patterns — the seven you must know

Recurring solutions to recurring OOP problems. Same shape, different names across teams.

Design pattern = a named, reusable solution to a problem that keeps showing up in OOP. The Gang of Four book lists 23. You only need to know around 7 for most interviews and real work.

The three families

BehaviouralHow objects communicate.Strategy, Observer

1. Singleton — only one of me, ever

Idea. Force a class to have exactly one instance, accessible globally. Used for things that don't make sense to duplicate: app config, a logger, a connection pool.

Real example. The Swiggy app has one AppConfig object that holds the user's city, language and theme. Every screen reads from the same object.

class AppConfig:
    _instance = None
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.city = "Bangalore"
        return cls._instance

a = AppConfig()
b = AppConfig()
print(a is b)        # True — same object
Careful

Singletons are global state in disguise. They make testing hard. Use only when there is a real reason for one instance.

2. Factory — ask, don't construct

Idea. A separate class or method decides which concrete class to create. Calling code says what it wants, not how to build it.

Real example. Uber needs to create the right Vehicle object based on the ride type the user picked.

class Vehicle:                   pass
class Bike(Vehicle):             pass
class Sedan(Vehicle):            pass
class Auto(Vehicle):             pass

class VehicleFactory:
    @staticmethod
    def create(ride_type: str) -> Vehicle:
        if ride_type == "bike":  return Bike()
        if ride_type == "auto":  return Auto()
        return Sedan()

v = VehicleFactory.create("auto")

3. Builder — build complex objects step by step

Idea. When an object has many optional fields, use a builder instead of a 12-parameter constructor.

Real example. A Swiggy Order with items, address, coupon, tip, instructions, payment method.

order = (OrderBuilder()
            .for_customer("Anita")
            .add_item("Margherita", 1)
            .add_item("Coke", 2)
            .with_coupon("FIRST50")
            .tip(20)
            .pay_with("UPI")
            .build())

4. Adapter — make incompatible classes work together

Idea. Wrap an existing class so its interface matches what your code expects. Like a plug adapter for a UK socket.

Real example. Your app's PaymentGateway expects charge(amount). You integrate Stripe, whose SDK exposes createCharge(amountInCents, currency). Wrap it.

class StripeAdapter(PaymentGateway):
    def __init__(self, stripe_sdk): self.sdk = stripe_sdk
    def charge(self, amount):
        return self.sdk.createCharge(int(amount * 100), "INR")

5. Decorator — add features without changing the class

Idea. Wrap an object with another object that adds behaviour. Chain wrappers to stack features.

Real example. A Starbucks Coffee. Wrap it with Milk, then with Sugar, then with Whip. Each wrapper adds to cost() and description().

class Coffee:
    def cost(self): return 100
    def desc(self): return "Coffee"

class Milk:
    def __init__(self, drink): self.drink = drink
    def cost(self): return self.drink.cost() + 30
    def desc(self): return self.drink.desc() + " + milk"

class Sugar:
    def __init__(self, drink): self.drink = drink
    def cost(self): return self.drink.cost() + 10
    def desc(self): return self.drink.desc() + " + sugar"

order = Sugar(Milk(Coffee()))
print(order.desc(), order.cost())     # Coffee + milk + sugar  140

6. Strategy — swap an algorithm at runtime

Idea. Encapsulate a family of algorithms behind one interface. The owner object holds a reference to the strategy and calls it. Swap strategies without touching the owner.

Real example. Uber surge pricing. The Trip object holds a PricingStrategy. In peak hours, set it to SurgePricing. In off-hours, BasePricing. Trip never knows the difference.

class PricingStrategy:
    def price(self, km, mins): ...
class BasePricing(PricingStrategy):
    def price(self, km, mins): return 50 + 12*km + 2*mins
class SurgePricing(PricingStrategy):
    def price(self, km, mins): return (50 + 12*km + 2*mins) * 1.8

class Trip:
    def __init__(self, strat): self.strat = strat
    def fare(self, km, mins):  return self.strat.price(km, mins)

You already saw this in topic 30 (Designing Uber).

7. Observer — tell me when something changes

Idea. One object (the subject) keeps a list of observers. When its state changes, it notifies them. Used in event-driven systems.

Real example. A YouTube Channel has subscribers. When a new video is uploaded, the channel notifies every subscriber. Same shape: notifications, UI re-renders, stock-price tickers, Slack-bot triggers.

class Channel:
    def __init__(self, name):
        self.name = name
        self.subscribers = []
    def subscribe(self, user):  self.subscribers.append(user)
    def upload(self, video):
        for u in self.subscribers:
            u.notify(self.name, video)

class User:
    def __init__(self, name): self.name = name
    def notify(self, channel, video):
        print(self.name, "got: new video from", channel, "→", video)

ch = Channel("MKBHD")
ch.subscribe(User("Anita"))
ch.subscribe(User("Rohit"))
ch.upload("iPhone 17 review")

Pattern names are vocabulary

Even if you never explicitly “use” them, knowing the names lets you talk to other engineers in shorthand. “Wrap it in an adapter” saves a paragraph of explanation.

Don't pattern-hunt

Bad junior code is when somebody learns patterns and tries to use one everywhere. Patterns solve specific problems. If your problem is simple, simple code is the right answer.
