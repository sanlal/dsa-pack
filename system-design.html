Low-Level Design classes, objects, patterns
01 OOP foundations Classes, objects, the four pillars, and everything an interviewer can ask — explained from scratch. basic18 min ▾
TL;DR OOP turns code from "loose variables and far-away functions" into objects that own their data and the rules to change it. The four pillars — encapsulation, inheritance, polymorphism, abstraction — are the four habits that keep those objects safe, reusable and swappable. Interviewers don't want textbook definitions; they want to see you use them when you design a class.

The scenario — why OOP had to be invented

Imagine you're writing a banking app the old way, with just variables and functions. To track 1,000 customers you carry 1,000 sets of loose variables: balance1, name1, type1, balance2, name2, type2 … Every deposit, every withdrawal, every interest run lives in a helper function that takes all those variables as arguments.

Now your manager says "add a currency field". You go edit fifty functions. Then "validate that deposit amounts are positive" — fifty more edits. One forgotten edit and someone deposits -100 and steals money from the bank.

OOP's answer is simple: put the balance, the validation, and the operations inside one box called an object. The outside world just says account.deposit(100) and the object enforces its own rules. The data lives next to the code that owns it. Change the rule once, every caller is protected.

What OOP actually is — three angles

Textbook

"A programming paradigm built around objects: bundles of data (fields) and behaviour (methods) that interact by sending messages."

Engineer

"A way of organising code so that data and the rules that protect it travel together. The outside world calls methods; the object decides how to do them."

Interviewer

"When I ask you to design a parking lot or a chess game, how do you decompose it into classes? OOP is the toolkit; SOLID (next lesson) is the discipline."

Class vs Object — the cookie-cutter analogy

A class is the cookie cutter — the blueprint. It says "every cookie has a shape and a flavour." An object (also called an instance) is one actual cookie — chocolate, star-shaped, baked at 3 pm. You write one class once; you create as many objects from it as you like, and each object carries its own copy of the fields.

class Cookie {                              // the cutter (blueprint)
public:
    string flavour;
    string shape;
    Cookie(string f, string s) : flavour(f), shape(s) {}
};

Cookie a("chocolate", "star");              // object #1
Cookie b("vanilla",   "heart");             // object #2
// a and b are independent. a.flavour does not affect b.flavour.
class Cookie {                              // blueprint
    String flavour, shape;
    Cookie(String f, String s) { flavour = f; shape = s; }
}

Cookie a = new Cookie("chocolate", "star");
Cookie b = new Cookie("vanilla",   "heart");
class Cookie:                               # blueprint
    def __init__(self, flavour, shape):
        self.flavour = flavour
        self.shape   = shape

a = Cookie("chocolate", "star")
b = Cookie("vanilla",   "heart")

The anatomy of a class

Every class is made of the same handful of parts. Learn the vocabulary once.

PartWhat it isExample

Access modifiers — who is allowed in?

Modifiers control which code can touch which member. They are how OOP enforces boundaries.

ModifierVisible toUse for
publicEveryone.Methods that form the class's contract.
privateOnly this class.Fields and helper methods. Default for state.
protectedThis class + subclasses.State that subclasses legitimately need.
package / internalSame package / module.Java's default. Use for collaborators that ship together.

Python doesn't enforce these — a leading _ means "treat as private", and __ triggers name-mangling. C++ has all three; Java has all four; Python relies on convention.

Static vs instance members — the "one per class" rule

Instance members live inside each object (one balance per account). Static members live on the class itself (one bank-wide tax rate, shared by everyone). Use static for constants, counters, utility functions, and factories.

class BankAccount {
public:
    static double TAX_RATE;                 // one for everyone
    double balance;                         // one per object
    static int count() { return total_; }   // utility
private:
    static int total_;
};
double BankAccount::TAX_RATE = 0.05;
int    BankAccount::total_   = 0;
class BankAccount {
    public static final double TAX_RATE = 0.05;   // shared constant
    private double balance;                       // per object
    private static int total = 0;                 // shared counter
    public static int count() { return total; }
}
class BankAccount:
    TAX_RATE = 0.05                # class attribute (shared)
    _total   = 0
    def __init__(self, b):
        self.balance = b           # instance attribute (per object)
    @classmethod
    def count(cls):
        return cls._total

Constructors in depth

The constructor is the only place where an object goes from "doesn't exist" to "ready to use". A few flavours show up in interviews.

  • Default constructor — takes no arguments. Most languages generate one for free if you write none yourself.
  • Parameterised constructor — takes arguments to set initial state.
  • Copy constructor (mostly C++) — takes another object of the same type and clones it. Crucial when your class owns raw pointers.
  • Constructor chaining — one constructor calls another (this(…) in Java, delegating constructors in C++11+, default arguments in Python). Removes duplicate setup.
  • Constructor calling its parent — super(…) in Java/Python, base initialiser list in C++. Must be the first thing the child constructor does.
  • Static factory method — a named static method that creates and returns an instance. Reads better than a long constructor: User.fromEmail("a@b.com").

Object lifecycle — from new to gone

  1. Allocation. Memory is reserved for the object's fields.
  2. Construction. The constructor runs — parent first, then child. Fields get their initial values.
  3. Use. Callers send messages (call methods). The object reads and updates its own state.
  4. Destruction. In C++ the destructor runs deterministically when the object goes out of scope (RAII). In Java and Python a garbage collector reclaims memory whenever it likes — so close files, sockets and DB connections explicitly (try-with-resources, with statement). Never rely on a finaliser.
The four pillars deep enough to defend in an interview

Pillar 1 — Encapsulation

What it is. Hide the fields. Expose methods. Inside those methods, validate every change so the object never enters an invalid state. Encapsulation is the difference between "a balance" (just data) and "a bank account" (data that polices itself).

Real-life analogy. An ATM. You can't reach inside and change the cash drawer directly — you press buttons (methods). The ATM checks your PIN, your balance, the daily limit (validation), and only then performs the action.

class BankAccount {
    double balance_;                          // hidden
public:
    explicit BankAccount(double opening) : balance_(opening) {
        if (opening < 0) throw runtime_error("opening must be >= 0");
    }
    void deposit(double amt) {
        if (amt <= 0) throw runtime_error("deposit must be positive");
        balance_ += amt;
    }
    void withdraw(double amt) {
        if (amt <= 0)           throw runtime_error("bad amount");
        if (amt > balance_)     throw runtime_error("insufficient funds");
        balance_ -= amt;
    }
    double balance() const { return balance_; }   // read-only getter
};
public class BankAccount {
    private double balance;                       // hidden
    public BankAccount(double opening) {
        if (opening < 0) throw new IllegalArgumentException("opening < 0");
        this.balance = opening;
    }
    public void deposit(double amt) {
        if (amt <= 0) throw new IllegalArgumentException("amt <= 0");
        balance += amt;
    }
    public void withdraw(double amt) {
        if (amt <= 0)         throw new IllegalArgumentException("amt <= 0");
        if (amt > balance)    throw new IllegalStateException("no funds");
        balance -= amt;
    }
    public double getBalance() { return balance; }   // no setter
}
class BankAccount:
    def __init__(self, opening: float):
        if opening < 0:
            raise ValueError("opening < 0")
        self._balance = opening                   # _ = hands off

    def deposit(self, amt: float):
        if amt <= 0: raise ValueError("amt <= 0")
        self._balance += amt

    def withdraw(self, amt: float):
        if amt <= 0:                raise ValueError("amt <= 0")
        if amt > self._balance:     raise ValueError("no funds")
        self._balance -= amt

    @property
    def balance(self):                            # read-only
        return self._balance
Getter / Setter rules Expose a getter if the outside world genuinely needs to read the field. Expose a setter only if it makes sense to change after construction — and even then, validate inside it. Better still: expose behaviour (deposit, withdraw), not raw setters. A class that's all getters and setters has no encapsulation; it's just a struct in disguise.
Bonus: immutability An object whose fields can never change after construction is immutable. Strings in Java and Python are immutable. Immutable objects are thread-safe for free and impossible to corrupt — use them whenever you can (configs, value objects, money amounts).

Pillar 2 — Inheritance

What it is. A child class reuses the fields and methods of a parent class and adds or overrides what it needs. Models the "is-a" relationship.

Real-life analogy. A SavingsAccount is a BankAccount — it has everything a BankAccount has, plus the extra concept of an interest rate. You don't rewrite deposit and withdraw; you inherit them.

class SavingsAccount : public BankAccount {
    double rate_;
public:
    SavingsAccount(double opening, double rate)
        : BankAccount(opening), rate_(rate) {}       // base ctor first
    void addInterest() { deposit(balance() * rate_); }
};
public class SavingsAccount extends BankAccount {
    private final double rate;
    public SavingsAccount(double opening, double rate) {
        super(opening);                              // must be first line
        this.rate = rate;
    }
    public void addInterest() { deposit(getBalance() * rate); }
}
class SavingsAccount(BankAccount):
    def __init__(self, opening, rate):
        super().__init__(opening)
        self.rate = rate
    def add_interest(self):
        self.deposit(self.balance * self.rate)

The diamond problem

If B and C both inherit from A, and D inherits from both B and C, then D seemingly has two copies of A's fields. Which one wins?

  • C++ — you must mark A as a virtual base (class B : virtual public A) to get a single shared copy.
  • Java — sidesteps it by banning multiple class inheritance. You can implement many interfaces, and if two interfaces have a default method with the same signature the compiler forces you to override it.
  • Python — resolves it using the MRO (Method Resolution Order, C3 linearisation). Inspect with D.__mro__. Predictable, but worth understanding before you stack multiple inheritance.

Sealing the chain

You can stop further inheritance with final class (Java), sealed (modern Java / Kotlin), or just by convention in C++/Python. Sealing is useful when subclassing would break invariants (e.g. String is final in Java).

Don't reach for inheritance first Inheritance is the tightest possible coupling: the child knows the parent's internals. If the relationship isn't a clean, permanent "is-a", prefer composition (hold a reference to a collaborator). The mantra is "favour composition over inheritance" — the moment you find yourself overriding methods to undo the parent's behaviour, you've picked the wrong tool.

Pillar 3 — Polymorphism

What it is. One name, many forms. The same call (shape.area()) dispatches to a different implementation depending on the actual object's type at runtime. Lets you write code against an abstraction and have it work with any concrete subclass — including ones written years later.

Two flavours

Compile-time (static)

Method overloading — same name, different parameter lists. The compiler picks which one to call based on the arguments you pass.

int   sum(int a, int b);
double sum(double a, double b);
int   sum(int a, int b, int c);

Also includes operator overloading in C++ and Python (__add__, __eq__). Java does not allow operator overloading.

Runtime (dynamic)

Method overriding — a child class redefines a method declared in the parent. The actual call is resolved at runtime based on the object's real type, not the variable's declared type.

This is the polymorphism that powers plugin systems, the Strategy pattern, and most of the GoF design patterns. Achieved via virtual functions in C++, all non-static methods in Java/Python by default.

struct Shape {
    virtual double area() const = 0;          // pure virtual
    virtual ~Shape() = default;               // virtual dtor!
};
struct Circle : Shape {
    double r;
    explicit Circle(double r) : r(r) {}
    double area() const override { return 3.14159 * r * r; }
};
struct Square : Shape {
    double s;
    explicit Square(double s) : s(s) {}
    double area() const override { return s * s; }
};

double totalArea(const vector<Shape*>& shapes) {
    double t = 0;
    for (auto* s : shapes) t += s->area();   // dispatch decided at runtime
    return t;
}
abstract class Shape { abstract double area(); }

class Circle extends Shape {
    private final double r;
    Circle(double r) { this.r = r; }
    @Override double area() { return Math.PI * r * r; }
}
class Square extends Shape {
    private final double s;
    Square(double s) { this.s = s; }
    @Override double area() { return s * s; }
}

double totalArea(List<Shape> shapes) {
    double t = 0;
    for (Shape s : shapes) t += s.area();   // virtual call
    return t;
}
from abc import ABC, abstractmethod

class Shape(ABC):
    @abstractmethod
    def area(self): ...

class Circle(Shape):
    def __init__(self, r): self.r = r
    def area(self): return 3.14159 * self.r * self.r

class Square(Shape):
    def __init__(self, s): self.s = s
    def area(self): return self.s * self.s

def total_area(shapes):
    return sum(s.area() for s in shapes)    # duck typing + dispatch
How runtime dispatch actually works Most compilers implement runtime polymorphism with a vtable — each class has a hidden table of function pointers, and every object carries a pointer to its class's table. Calling shape.area() looks up the correct function via the vtable. In Python the lookup walks the object's class chain (the MRO). The cost is one pointer indirection — cheap, but not free, which is why C++ asks you to opt in via virtual.
Always make destructors virtual when you have virtual methods (C++) Otherwise delete basePtr only runs the base destructor and leaks the derived part. Java and Python don't have this trap because they don't have destructors.

Pillar 4 — Abstraction

What it is. Define what something does; defer how. Callers depend on a stable interface and remain unaware of (and unaffected by) implementation choices.

Real-life analogy. A car's steering wheel. You turn it left, the car goes left. You don't know — or care — whether the car uses hydraulic, electric, or steer-by-wire underneath. Replace the engine; the wheel still works.

Abstract class vs Interface — the eternal interview question

Can have implementation?Yes — concrete methods alongside abstract ones.Mostly no. Java 8+ allows default methods.
How many can a class extend / implement?One (in Java).Many.
Constructor?Yes — called by subclass via super.No.

In C++ / PythonClass with at least one pure-virtual method (C++) / ABC with @abstractmethod (Python).No separate keyword. Use a class with only pure-virtual methods (C++) or a Protocol / ABC (Python).

Rule of thumb: if you'd describe the relationship as "X is a Y" and they share data, use an abstract class. If you'd describe it as "X can do Y", use an interface. Bird is a Animal (abstract class). Bird can do Flyable (interface). A Penguin is a Bird that does not implement Flyable.

The five ways objects can relate

Knowing the difference is what separates a junior LLD answer from a senior one.

The plumbing small habits that mark a senior

Equality, hashing, and string form

Every value-like object should answer three questions correctly.

  • Equality. Are these two objects "the same"? Override equals (Java), operator== (C++), __eq__ (Python). Default equality compares references, not contents — almost never what you want.
  • Hashing. If two objects are equal, their hash codes must match. Override hashCode (Java) / __hash__ (Python) every time you override equality, or your objects break inside HashMap/HashSet.
  • String form. Override toString (Java) / __str__ + __repr__ (Python) / operator<< (C++) so logs and debuggers show something useful.

Shallow vs deep copy

A shallow copy duplicates the object's top-level fields; references inside still point to the same shared things. A deep copy recursively duplicates everything. In Python, copy.copy() vs copy.deepcopy(); in Java, override clone() (or use a copy constructor — usually cleaner); in C++, write a copy constructor and copy-assignment operator that follow the rule of three / five.

Pitfalls that show up in interviews

Avoid

God class — one class that does everything. Always smell, always wrong.

Public fields — no encapsulation, no validation, breaking change risk.

Inheriting just to reuse code — if it's not "is-a", compose.

Overriding to remove behaviour — classic LSP violation (next lesson).

Calling overridable methods from a constructor — the child's override may run before the child's fields are initialised. Subtle bug.

Forgetting virtual ~Base() in C++ when you have virtual methods.

Do

Small classes — one reason to exist.

Private fields, behaviour-style methods — deposit(), not setBalance().

Program to interfaces — depend on List, not ArrayList.

Make value objects immutable — money, dates, coordinates.

Prefer composition — reach for inheritance only when the "is-a" is permanent and clean.

Honour the equals/hashCode contract whenever you store objects in hash-based collections.

Quick recap — three lines

  1. Object = data + the rules to change it. Hide fields, expose behaviour, validate on every change.
  2. The four pillars are habits, not theory. Encapsulation hides, inheritance reuses, polymorphism swaps, abstraction stabilises the contract.
  3. Favour composition over inheritance. If the relationship isn't a clean, permanent "is-a", hold a reference instead of subclassing.
How to answer "what is OOP?" in 30 seconds "OOP organises code around objects — bundles of data and the methods that operate on them. The four pillars are encapsulation (hide state, expose behaviour), inheritance (model is-a, but use sparingly), polymorphism (one call, many implementations — swap behaviour at runtime), and abstraction (program against contracts, not concretes). The whole point is to keep change local: when a requirement shifts, only one class should need to change." — then pivot to a concrete example from your design.
02 SOLID principles Five rules that turn fragile code into flexible code. intermediate7 min ▾
TL;DR SOLID = SRP, OCP, LSP, ISP, DIP. Each one prevents a specific kind of pain. Memorise the pain, not just the acronym.

S — Single Responsibility Principle

"A class should have one reason to change." A User class that holds data, saves to DB, and sends emails has three reasons to change — split it.

Bad

class User {
  save() {...}
  sendWelcomeEmail() {...}
  formatForCsv() {...}
}

Good

class User { /* data only */ }
class UserRepository { save(u) {...} }
class WelcomeMailer { send(u) {...} }
class UserCsv { format(u) {...} }

O — Open/Closed

"Open for extension, closed for modification." Adding a new payment method shouldn't require editing the PaymentProcessor. Use polymorphism (Strategy pattern).

Bad

switch(method) {
  case CARD: chargeCard(); break;
  case UPI:  chargeUpi();  break;
  // adding PayPal? edit here, retest everything
}

Good

interface PaymentMethod { void charge(Order o); }
class Card implements PaymentMethod {...}
class Upi  implements PaymentMethod {...}
// add PayPal class without touching old code

L — Liskov Substitution

A subclass should be usable wherever its parent is — without surprises. The classic violation: Square extends Rectangle and overrides setWidth to also change height. Now any code expecting a Rectangle behaves wrong with a Square.

I — Interface Segregation

"Many small interfaces beat one giant one." A printer-scanner shouldn't force a plain printer to implement scan(). Split into Printer and Scanner, and let the multi-function device implement both.

D — Dependency Inversion

"Depend on abstractions, not concrete classes." High-level OrderService should depend on a PaymentGateway interface, not on StripeClient. Inject the concrete implementation at construction time. Now testing with a fake gateway is trivial.

class PaymentGateway { public: virtual bool charge(double) = 0; };

class OrderService {
    PaymentGateway& gw;                    // injected
public:
    OrderService(PaymentGateway& g) : gw(g) {}
    bool place(double amt) { return gw.charge(amt); }
};
interface PaymentGateway { boolean charge(double amt); }

class OrderService {
    private final PaymentGateway gw;       // injected
    OrderService(PaymentGateway gw) { this.gw = gw; }
    boolean place(double amt) { return gw.charge(amt); }
}
from typing import Protocol
class PaymentGateway(Protocol):
    def charge(self, amt: float) -> bool: ...

class OrderService:
    def __init__(self, gw: PaymentGateway):
        self.gw = gw
    def place(self, amt):
        return self.gw.charge(amt)
Takeaway SOLID isn't dogma — it's a checklist when you smell bad code. "Does this class do too much? (SRP) Does this switch grow every release? (OCP)" The right principle usually pops out.
03 Design patterns interviewers love Singleton, Factory, Builder, Observer, Strategy, Decorator, Adapter. intermediate10 min ▾
TL;DR Design patterns are names for solutions you'd discover anyway. Learn the 7 below and you'll spot them — and apply them — in any LLD round.

Singleton — exactly one instance

Use for shared config, loggers, connection pools. Make the constructor private; expose a static getInstance(). Beware: untestable, hides dependencies, hard to scale to multi-threaded.

class Logger {
    Logger() {}
public:
    static Logger& instance() {        // Meyers' singleton, thread-safe since C++11
        static Logger inst;
        return inst;
    }
    void log(const string& s) { /* ... */ }
};
public class Logger {
    private static volatile Logger INSTANCE;
    private Logger() {}
    public static Logger getInstance() {
        if (INSTANCE == null) {
            synchronized (Logger.class) {
                if (INSTANCE == null) INSTANCE = new Logger();
            }
        }
        return INSTANCE;
    }
}
class Logger:
    _inst = None
    def __new__(cls):
        if cls._inst is None:
            cls._inst = super().__new__(cls)
        return cls._inst

Factory — create without naming the class

Callers ask the factory for "a parser for JSON"; factory returns the right concrete type. Adding a new format = one new class + one line in the factory.

Builder — fluent construction of complex objects

When a constructor would take 10 parameters, replace it with a builder. new Pizza.Builder().crust(THIN).cheese().pepperoni().build().

Observer — publish/subscribe

Subject keeps a list of observers; on state change, notifies them all. Backbone of UI event handlers, Kafka consumers, the Redux store.

class Observer { public: virtual void on(const string& ev) = 0; };
class Subject {
    vector<Observer*> subs;
public:
    void subscribe(Observer* o) { subs.push_back(o); }
    void publish(const string& ev) { for (auto* o : subs) o->on(ev); }
};
interface Observer { void on(String ev); }
class Subject {
    private final List<Observer> subs = new ArrayList<>();
    void subscribe(Observer o) { subs.add(o); }
    void publish(String ev) { for (Observer o : subs) o.on(ev); }
}
class Subject:
    def __init__(self):
        self.subs = []
    def subscribe(self, fn):
        self.subs.append(fn)
    def publish(self, ev):
        for fn in self.subs:
            fn(ev)

Strategy — swap the algorithm at runtime

Sort with comparator, payment with PaymentMethod, compression with Compressor. Strategy is OCP in code form.

Decorator — wrap to add behaviour

Coffee → MilkDecorator(Coffee) → SugarDecorator(MilkDecorator(Coffee)). Each wrapper adds price/description without modifying the original.

Adapter — bridge incompatible interfaces

Your app expects a Logger.log(s); the third-party library exposes RemoteLogger.send(msg, level). An adapter class translates the call.

How to use patterns in interviews Don't shout "Observer pattern!". Describe the design ("the cart will notify subscribers when totals change") and the interviewer will recognise the pattern. Naming feels show-off; describing feels confident.
Takeaway 7 patterns cover 80% of LLD interviews. Internalise the problem each one solves and you'll reach for it naturally.
04 LLD walkthroughs Parking Lot, Splitwise, BookMyShow — the classics, fully sketched. case12 min ▾
TL;DR LLD interviewers want to see: (1) you ask clarifying questions, (2) you list nouns → classes, (3) you draw interactions, (4) you mention concurrency, (5) you handle one extension cleanly.

Approach for any LLD problem (5 steps)

  1. Clarify scope. What's in / what's out? E.g. "Is payment in scope, or only ticket issuance?"
  2. List entities (nouns). ParkingLot, Spot, Vehicle, Ticket, Payment.
  3. Define relationships. A ParkingLot has many Spots. A Ticket belongs to one Vehicle.
  4. Methods on each. ParkingLot.park(vehicle) -> Ticket, ParkingLot.leave(ticket) -> Receipt.
  5. Discuss concurrency & extensions. Two cars enter at the same time. Multi-floor lots. EV-only spots.

Walkthrough 1 — Parking Lot

Clarify: "Multi-level? Vehicle types? Pricing model? Reservation?". Assume: multi-floor, 3 vehicle types (bike/car/truck), pay-on-exit.

Classes

  • VehicleType enum: BIKE, CAR, TRUCK.
  • Vehicle (plate, type).
  • Spot (id, type, isFree).
  • Floor (number, spots, findFreeSpot(type)).
  • ParkingLot (floors, park(), leave(), registry of active tickets).
  • Ticket (id, vehicle, spot, entryTime).
  • PricingStrategy interface: HourlyPricing, FlatPricing (Strategy pattern).
  • Payment with state machine: PENDING → PAID / FAILED.

Concurrency: two cars contend for the same spot. Make findFreeSpot + reserve atomic (lock the floor, or use a BlockingQueue of free spots per type).

Extension scenario: "Now we support EV charging spots." Add EvSpot extends Spot + a new VehicleType.EV. findFreeSpot(EV) can fall back to a regular CAR spot if no EV spot is free — document the policy.

Walkthrough 2 — Splitwise (expense splitter)

Clarify: "Group expenses? Equal split / percentage / exact / shares? Settlement suggestions?"

Classes

  • User (id, name, email).
  • Group (members, expenses).
  • Expense (paidBy, totalAmount, splits, splitType).
  • Split abstract: EqualSplit, PercentageSplit, ExactSplit (Strategy).
  • BalanceSheet: Map<User, Map<User, Double>> = "owes how much to whom".
  • SettlementService.minimiseTransactions(group): turn the balance graph into the fewest transfers (greedy + max-heap of creditors/debtors).

Concurrency: two members add expenses to the same group simultaneously → serialise updates to the BalanceSheet (lock per group).

Walkthrough 3 — BookMyShow (movie ticket)

Clarify: "Single city / multi-city? Concurrent seat selection? Refunds?"

Classes

  • City, Theatre, Screen, Show (movie, screen, startTime, seatLayout).
  • Seat (row, col, type: REGULAR / RECLINER / GOLD).
  • Booking (user, show, seats, status: HOLD / CONFIRMED / CANCELLED, expiresAt).
  • SeatHold with TTL (e.g. 5 minutes) so abandoned carts release seats.
  • Payment + BookingService.confirm(holdId, paymentToken).

The hard part — two users tap the same seat at the same time. Wrap seat selection in a transaction: UPDATE seats SET held_by = ?, expires = ? WHERE show_id = ? AND seat_id IN (...) AND held_by IS NULL. If 0 rows updated → someone else got it; show the user the new seat map.

Don't draw God classes BookMyShow with 200 methods is a red flag. Decompose: CatalogService, SeatService, BookingService, PaymentService, NotificationService.
Takeaway The LLD interview rewards narrating your design as you build it. Talk through the entities, drop in 1-2 patterns, mention concurrency. You're done in 30 minutes.

05 Scalability vocabulary The 10 words you must use correctly in HLD rounds. basic5 min ▾
TL;DR If you mix up latency and throughput, or SLA and SLO, the interviewer mentally notes "junior". Get these definitions perfect.
TermMeaning in one lineExample number
Latency Time for one request to finish. 50 ms p50, 200 ms p99
Throughput / QPS Requests served per second. 10k QPS
Vertical scaling Bigger box: more CPU / RAM / disk. 2x cores
Horizontal scalingMore boxes behind a load balancer. 3 → 30 nodes
Availability Fraction of time the system answers correctly. 99.99% = 52 min/year down
Durability Probability data survives. S3 = 11 nines
SLA Contractual promise to customer (legally binding). "99.9% uptime or refund"
SLO Internal target you measure against. "p99 latency < 300 ms"
SLI The actual metric you're measuring. "% of requests under 300 ms"
Idempotent Calling N times = calling once. HTTP PUT, DELETE

Vertical vs horizontal — when to choose what

Vertical scaling

+ Zero code change.

+ Best for state-heavy single nodes (a primary DB).

- Hard cap (you run out of biggest box).

- Single point of failure.

Horizontal scaling

+ Near-infinite ceiling.

+ Fault tolerant: lose 1 of 50 = ok.

- Needs stateless services (or sharding).

- Coordination + distributed-systems pain.

Takeaway "Scale up first, scale out next" is a fine rule of thumb. Start vertical, switch to horizontal when latency or HA forces you.
06 CAP theorem & consistency The triangle you can never fully escape. theory6 min ▾
TL;DR A distributed system can only guarantee two of {Consistency, Availability, Partition tolerance} during a network split. Partitions will happen, so the real trade-off is C vs A.
CP systems halt writes during split; AP systems answer with stale data.

Consistency models (from strong to loose)

  • Strong / linearizable. Every read sees the latest write. Slow, expensive, but easy to reason about (single MySQL primary).
  • Read-your-writes. After you write, you see it; others may lag. Sticky sessions, or read from primary for your user.
  • Monotonic reads. Once you've seen version N, you never see version < N again.
  • Eventual. All replicas converge "eventually". Cheap, fast, but reads can return stale data (DynamoDB by default, Cassandra).
  • Causal. Operations that depend on each other are seen in order; unrelated ops can reorder.

How to talk about it in an interview

Don't say "CAP theorem!" abstractly. Say: "For a banking system, I'd pick CP — we can't show a stale balance. For a social feed, AP is fine — a 5-second-stale like count won't ruin anyone's day."

PACELC is the modern refinement: If Partition, choose A or C; Else, choose Latency or Consistency. Spanner is CP+EL (sacrifices latency for consistency even with no partition). Cassandra is AP+EL.
Takeaway "We need strong consistency" is rarely true. Most domains tolerate eventual consistency — and unlock huge scale by accepting it.
07 Load balancing Spreading traffic without losing your mind. intermediate7 min ▾
TL;DR A load balancer sits between clients and a fleet of servers, distributing requests. Two big choices: L4 vs L7, and the algorithm (round-robin, least-connections, consistent hashing).
One brain, many backs.

L4 vs L7

L4 (transport)

Routes by IP + port, ignores payload. Fast, simple, cheap.

Tools: AWS NLB, IPVS, HAProxy in TCP mode.

L7 (application)

Reads HTTP. Can route by path, header, cookie. Can do TLS termination, retries, rate limiting.

Tools: AWS ALB, NGINX, Envoy, Istio.

Algorithms

  • Round-robin. req 1 → A, req 2 → B, req 3 → C, req 4 → A. Simple but ignores load.
  • Least connections. Send to the server with fewest active connections. Better for variable-length requests.
  • Weighted. Bigger servers get more weight. Useful during a rolling deploy.
  • IP hash. Same client always lands on the same server (cheap sticky session).
  • Consistent hashing. Adding/removing a server only re-maps ~1/N of keys instead of all. Critical for cache fleets and sharded systems.

Health checks & failover

The LB pings each backend on a port/path every few seconds. If a backend fails k consecutive checks, it's removed from rotation. Re-added when healthy again. Crucial: don't share the LB itself as a single point of failure — run multiple LBs behind DNS round-robin or BGP anycast.

Takeaway Default to L7 for HTTP apps — the routing flexibility pays off. Use consistent hashing for any cache or sharded backend so node changes don't trigger thundering re-balancing.
08 Caching layers Every cache hit is a query the DB didn't see. intermediate7 min ▾
TL;DR Cache aggressively, invalidate carefully. The cache always wins on read performance; the cost is on write and on staleness.
The closer to the user, the faster — and the staler.

Five layers worth caching

  1. Browser. Cache-Control headers, service workers, IndexedDB.
  2. CDN. Cloudflare, Akamai, CloudFront. Caches your static assets near the user (~10 ms).
  3. App-level / in-process. LRU map inside the server process. Lowest latency, lost on restart.
  4. Distributed cache. Redis or Memcached cluster. Shared across many app instances.
  5. DB cache. Buffer pool, query cache. Mostly automatic but tune sizes.

Write strategies

Write-through

Write to cache and DB together. Safe, but slower writes.

Write-back

Write to cache; flush to DB later. Fast writes; risky if cache dies.

Write-around

Write straight to DB; cache fills on next read. Fine for "write once, read rarely".

Eviction policies

  • LRU (least recently used) — default in Redis (with allkeys-lru). Works well for general traffic.
  • LFU (least frequently used) — resists "one-hit wonders" like a viral tweet polluting the cache.
  • TTL — expire after N seconds. Simple, predictable staleness.

Cache invalidation — "two hard problems"

  • Write-through updates the cache on every write → never stale, slower writes.
  • Cache-aside + TTL. Read populates cache; writes only invalidate (or delete) the key. The TTL caps how stale a missed invalidation can stay.
  • Pub/sub invalidation. Writer publishes a "user-123 changed" event; all app servers listen and drop the entry.
Thundering herd when a hot key expires and 10,000 requests hit the DB simultaneously. Mitigate with request coalescing (single re-fetch, others wait) or stale-while-revalidate (serve stale, refresh in background).
Takeaway If your read traffic to write traffic is >10:1, you should be caching. If it's >100:1, the cache is doing 99% of the work.
09 Databases: SQL, NoSQL, sharding Choose the storage that matches the question. advanced9 min ▾
TL;DR SQL for relational + ACID. NoSQL when you outgrow it or need flexible schema. Always-index your queries. Shard only when one box stops being enough.

SQL vs NoSQL at a glance

DimensionSQL (Postgres, MySQL)NoSQL
Schema Strict, enforced Flexible / schema-less
Joins First-class Usually denormalised, joins in app
Consistency ACID (strong) BASE (eventual) by default
Scale Mostly vertical + read replicasHorizontal-first, sharded
Best for Orders, payments, anything with FKsFeeds, logs, key-value, time series

NoSQL flavours (pick by access pattern)

  • Key-value (Redis, DynamoDB) — "given a key, get a value". Sessions, caches, profiles.
  • Document (MongoDB, Couchbase) — JSON-ish docs with nested fields and partial indexes.
  • Wide-column (Cassandra, HBase, ScyllaDB) — tunable consistency, write-heavy, time-series.
  • Graph (Neo4j, Amazon Neptune) — relationship-heavy queries (friend-of-friend).
  • Search (Elasticsearch, OpenSearch) — full-text and faceted search.

Indexing 101

A B-tree index turns "search column X" from O(n) into O(log n). Trade-off: each insert/update/delete must also update every index → slower writes. Rules of thumb:

  • Index the columns you query in WHERE, JOIN, ORDER BY.
  • Composite index (a, b) works for queries on a alone and (a, b), but not on b alone.
  • Never over-index a write-heavy table. Measure with EXPLAIN.

Replication

Leader-follower (primary-replica): writes go to leader; reads can hit followers (slight lag). Standard MySQL/Postgres setup.
Multi-leader: writes on any node; conflict resolution needed.
Leaderless (Cassandra): every node accepts writes; quorum reads/writes for consistency.

Sharding (horizontal partitioning)

Cheap reads-per-shard. Cross-shard queries get painful.
  • Hash sharding. shard = hash(user_id) % N. Even distribution; ranged queries cost a fan-out.
  • Range sharding. Users A-F → shard 0, G-M → shard 1. Range queries fast, but hot keys cause hotspots.
  • Geo sharding. By region. Useful for latency & data-locality laws (GDPR).
  • Consistent hashing. Add a shard → only ~1/N of keys move (vs ~all with mod N).
Don't shard early Sharding is a one-way door — cross-shard joins, transactions, and rebalances all become hard. Try vertical scaling + read replicas + caching first.
Takeaway Start with a single Postgres. Add a read replica. Add a cache. Shard only when you've exhausted the above — usually past 10TB of hot data.
10 Message queues & events Decouple producers and consumers; smooth out spikes. advanced7 min ▾
TL;DR A queue is a buffer between "I'm done" and "you take it from here". Producers and consumers scale independently, slow downstream doesn't break upstream, and you get retries for free.

Three use cases

  • Async work. User uploads a video; web server returns 202, video service processes it later from a queue.
  • Spike absorption. Flash sale: 100k orders/sec into the queue, 1k orders/sec processed downstream — queue stretches.
  • Decoupling. Order service publishes OrderPlaced; mail, analytics, fraud services each subscribe.

Kafka vs RabbitMQ vs SQS

ToolModelStrengthUse when…
Kafka Log (replayable) High throughput, retention event sourcing, analytics, audit
RabbitMQ Queue + exchanges Flexible routing, low latencyRPC-like task queues
SQS Managed queue Zero ops You're on AWS and want simple
Redis StreamsLightweight log If Redis is already there Small-scale events

Delivery guarantees (memorise!)

  • At-most-once — might lose messages, never duplicate. Fire-and-forget metrics.
  • At-least-once — never lose, may duplicate. The default in practice; consumers must be idempotent.
  • Exactly-once — the dream. Kafka offers it for the Kafka-to-Kafka path with transactions; end-to-end exactly-once across services is essentially impossible without idempotency keys.
Idempotency trick Producer attaches a UUID per logical event. Consumer keeps a "processed IDs" set (Redis, DB unique index). Duplicate → skip. Now at-least-once + idempotency = exactly-once effects.

Dead-letter queues (DLQ)

After N failed retries, the message goes to a DLQ instead of looping forever. Engineers triage the DLQ. Never auto-drop — you'll lose evidence of bugs.

Takeaway Add a queue the moment two services can answer the question "does this need to happen synchronously?" with "no". Latency drops, resilience climbs.
11 Microservices & API gateway When to split the monolith — and when not to. advanced7 min ▾
TL;DR Microservices trade internal complexity for independent deployment & team scaling. Worth it past ~50 engineers; usually overkill below.
Each service owns its DB and emits events.

Monolith vs Microservices

Monolith wins when

  • Small team (< 20 devs).
  • Domain is still being discovered.
  • You need cross-table transactions a lot.
  • You want simple deploys.

Microservices win when

  • Teams need to deploy independently.
  • Different services have wildly different scale or tech.
  • Blast radius of a bad deploy must be small.
  • Org chart maps to service boundaries (Conway's law).

The supporting cast

  • API gateway — one front door. Authn/authz, rate limiting, routing, request shaping.
  • Service discovery — how does orders find payments? DNS (Kubernetes service), Consul, Eureka.
  • Circuit breaker — if downstream fails fast for 50% of requests, stop calling it for 30s. Prevents cascading failure (Hystrix, resilience4j).
  • Distributed tracing — OpenTelemetry, Jaeger. One request ID flows through every service so you can see the waterfall.
  • Saga pattern — replaces cross-service transactions: each step has a compensating action ("if payment fails, cancel the order").
Distributed monolith If your microservices must all be deployed together, you've built the worst of both worlds. Aim for clear boundaries + async communication.
Takeaway Start with a modular monolith. Extract a service only when a boundary screams (different scale, different team, different lifecycle). Premature microservices wastes years.
12 Rate limiting & auth Keep bad actors out, keep good ones inside the budget. intermediate6 min ▾
TL;DR Rate limit by user/IP/API-key. Authenticate with sessions for browser apps, JWTs for stateless APIs, OAuth for third-party access.

Rate-limiting algorithms

  • Fixed window. "100 req per minute starting on the minute". Simple but allows 2x burst at the boundary.
  • Sliding window log. Store every request timestamp; count those within the window. Accurate, expensive memory.
  • Sliding window counter. Approximate by combining current + previous window. Cheap and accurate enough.
  • Token bucket. Bucket of size B; refilled at rate R per second. Request takes 1 token, drops if empty. Allows controlled bursts.
  • Leaky bucket. Token bucket's cousin; smooths burst into steady drain.

Token bucket sketch

struct TokenBucket {
    double tokens, capacity, refillPerSec;
    double lastFill;
    bool allow() {
        double now = nowSeconds();
        tokens = min(capacity, tokens + (now - lastFill) * refillPerSec);
        lastFill = now;
        if (tokens >= 1) { tokens -= 1; return true; }
        return false;
    }
};
class TokenBucket {
    double tokens, capacity, refillPerSec, lastFill;
    synchronized boolean allow() {
        double now = System.nanoTime() / 1e9;
        tokens = Math.min(capacity, tokens + (now - lastFill) * refillPerSec);
        lastFill = now;
        if (tokens >= 1) { tokens -= 1; return true; }
        return false;
    }
}
import time
class TokenBucket:
    def __init__(self, capacity, refill_per_sec):
        self.capacity = capacity
        self.refill = refill_per_sec
        self.tokens = capacity
        self.last = time.monotonic()
    def allow(self):
        now = time.monotonic()
        self.tokens = min(self.capacity, self.tokens + (now - self.last) * self.refill)
        self.last = now
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False

Auth choices

  • Session cookies. Server stores the session; client holds a cookie. Easy to invalidate, secure-by-default with HttpOnly + SameSite.
  • JWT (JSON Web Token). Signed payload; server doesn't need to look it up. Great for stateless APIs. Hard to revoke before expiry — keep TTL short (15 min) + refresh tokens.
  • OAuth 2.0. "Login with Google" flow. Authorisation server returns an access token your app uses to call user-owned APIs.
Don't roll your own crypto Use a library. Hash passwords with bcrypt/argon2 (never MD5/SHA-256 alone). Always TLS. Store secrets in a vault, never in source code.
Takeaway Apply rate limits at the edge (API gateway) and at the service for defence-in-depth. Pick the auth scheme by who the caller is — browser, partner API, or end-user via 3rd-party.
13 HLD walkthroughs URL shortener, chat, news feed — full designs in interview style. case14 min ▾
TL;DR Every HLD answer follows the same shape: clarify → estimate → data model → high-level diagram → deep-dive on 1-2 components → bottlenecks & trade-offs.

Universal HLD framework

  1. Clarify scope & SLAs. "Do we need analytics? Custom aliases? What's expected p99 latency?"
  2. Back-of-envelope estimation. Users, QPS, storage, bandwidth.
  3. Data model. Core entities + access patterns.
  4. High-level diagram. Client → LB → service → cache → DB → queue. Draw on whiteboard.
  5. Deep-dive. Pick the spiciest component (hashing scheme, sharding key, feed ranking) and go deeper.
  6. Bottlenecks & trade-offs. What breaks at 10x? Consistency choices. SPOFs.

Walkthrough A — URL shortener (bit.ly)

Clarify: "Custom aliases? Expiry? Analytics? How long are URLs (read-heavy clearly)?"

Estimate (assume):

  • Writes: 100M/month = ~40 writes/sec.
  • Read:write = 100:1 → 4k reads/sec, 100k peak.
  • Storage: ~500 bytes/record × 100M/month × 5 years ≈ 3 TB.

Data model: (short_key PRIMARY, long_url, owner, created_at, expires_at).

Diagram:

Read-heavy; cache the redirect, replicate the KV store.

Short-key generation:

  • Hash & truncate (MD5(long_url)[:7] in base62). Collisions possible → rehash with salt.
  • Counter / Snowflake. Central ID generator gives a monotonically-increasing 64-bit ID; encode to base62. Collision-free; predictable URLs (bad for security — mitigate by shuffling).
  • Pre-generated pool. Background job creates batches of unused keys; service grabs one per request. Decouples write path from key generation.

Bottlenecks: hot reads → CDN + Redis. Sharded KV store keyed by short_key. Analytics? Drop click events into Kafka → ClickHouse for queries.

Walkthrough B — Chat (WhatsApp-lite)

Clarify: "1-1 only or groups? Message delivery semantics? Online status?"

Estimate: 500M users, 50 msgs/user/day → ~290k msg/sec; storage ~500 GB/day if 200 B per msg + metadata.

Components:

  • Edge layer: WebSocket gateway (~1M connections per box, sticky LB on hash(user_id)).
  • Presence service: Redis with user-online TTL; heartbeats every 30s.
  • Message service: writes to Cassandra keyed by (conversation_id, msg_id ascending). Range-scan efficient for "load last 50 msgs".
  • Pub/sub: Kafka topic per shard; gateways subscribe and push to connected clients.
  • Offline delivery: Push via FCM/APNs when user not connected.

Hardest part - delivery guarantees. WebSocket can drop. Use a per-message ID; client ACKs on receipt; server keeps unacked for < N seconds; resend on reconnect. End-to-end encryption (E2EE) means server never sees plaintext — complicates search, simplifies privacy.

Walkthrough C — News feed (Instagram-lite)

Clarify: "Chronological or ranked? Followers can be in millions (celebrities)?"

The fundamental choice - fan-out:

Fan-out on write (push)

When user A posts, copy the post-id into the feed cache of every follower.

+ Read is O(1) — just read your feed key.

- Celebrity with 100M followers = 100M writes per post.

Fan-out on read (pull)

At read time, merge recent posts from everyone the user follows.

+ Cheap write.

- Read is O(followees), heavy for power users.

The hybrid (Instagram, Twitter): push for normal users, pull for celebrities at read time, merge the two lists. Best of both worlds.

Ranking: off-line model scores (user_id, post_id) -> affinity; the feed service pulls top-N candidate posts and re-ranks on read.

Storage: Posts in a sharded blob/KV store (user_id partition). Media in object store (S3) + CDN. Feeds in Redis sorted-set keyed by user_id, score=timestamp or rank.

Always close with trade-offs "I chose push for the common case because reads dominate. For celebrities, I'd switch to pull because the write amplification would crush the system." That last sentence wins offers.
Takeaway The HLD interview isn't a quiz on tools — it's a structured conversation. Memorise the framework, practice 5 walkthroughs out loud, and you'll never be lost.
